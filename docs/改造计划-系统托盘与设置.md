> 同时支持 Linux 和 Windows（不考虑 macOS）。

## 1. 概述

### 目标

1. **系统托盘**：右下角托盘图标，右键菜单可开关窗口、进入设置页面、退出程序
2. **显示优先级可配**：当 Claude Code / Reasonix / 媒体播放 三者同时有活动时，用户可以配置谁排前面
3. **空闲显示可配**：没有活动时，用户可选择显示「CPU + 内存」「仅 CPU」「仅内存」「不显示」
4. **样式主题可配**：用户可在多套预设主题间切换（至少：深色默认 / 浅色 / 半透明磨砂）

### 非目标

- 不实现 macOS 支持
- 不做实时热重载主题编辑器（只做预设切换）
- 不做拖拽重排优先级 UI（用下拉选择器即可）

### 配置存储

- 位置：`~/.config/timo/settings.json`（Linux）/ `%APPDATA%\timo\settings.json`（Windows）
- 格式：JSON，通过 Wails Service 暴露给前端读写
- 默认值在 Go 结构体中定义，首次启动时自动创建

## 3. 实施步骤详解

### Phase A：设置持久化后端

**新建文件：`settings.go`**（无 build tag，Linux + Windows 共用）

```go
// 配置结构体
type TimoSettings struct {
    DisplayPriority []string `json:"displayPriority"` // 优先级数组，如 ["ai", "media"]
    IdleDisplay     string   `json:"idleDisplay"`     // "all" | "cpu" | "mem" | "none"
    Theme           string   `json:"theme"`            // "dark" | "light" | "frosted"
}

// 默认值
func DefaultSettings() TimoSettings {
    return TimoSettings{
        DisplayPriority: []string{"ai", "media"},
        IdleDisplay:     "all",
        Theme:           "dark",
    }
}
```

**方法：**
- `LoadSettings() (TimoSettings, error)` — 从配置文件读取，文件不存在时返回默认值
- `SaveSettings(s TimoSettings) error` — 写入配置文件
- `GetSettingsPath() string` — 跨平台返回配置目录：Linux → `~/.config/timo/settings.json`，Windows → `%APPDATA%\timo\settings.json`

**新建文件：`settingsservice.go`**（Wails Service 包装）

```go
type SettingsService struct {
    mu       sync.Mutex
    settings TimoSettings
}

func (s *SettingsService) Get() TimoSettings    // 返回当前设置
func (s *SettingsService) Update(s TimoSettings) // 更新并持久化 + 触发事件
```

**注册到 `main.go`：**

```go
settingsService := NewSettingsService()
mainApp.RegisterService(application.NewService(settingsService))
```

**事件：** 设置更新时通过 `mainApp.Event.Emit("settings-updated", settings)` 通知前端。

### Phase B：系统托盘

**新建文件：`tray.go`**（无 build tag，但内部有条件编译）

需要解决一个问题：Linux 和 Windows 的托盘实现差异较大。

#### 方案选择

两个方案：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **A. Wails v3 SystemTray API** | 框架自带，跨平台，API 统一 | Wails v3 alpha 阶段，文档不全；需要确认 `application.NewSystemTray()` 在 Linux (Wayland) 下的稳定性 |
| **B. 原生托盘 + 条件编译** | 可控性强，不依赖 Wails alpha API | 两份实现，开发量大 |

**推荐：方案 A —— 先尝试 Wails v3 SystemTray API，若遇到阻塞再回落方案 B。**

#### Wails v3 SystemTray API 用法（参考框架源码）

```go
// tray.go
package main

func setupSystemTray(app *application.App, toggleWindow func(), openSettings func(), quit func()) {
    tray := app.NewSystemTray(&application.SystemTrayOptions{
        Icon:        appicon,  // 应用图标
        Tooltip:     "Timo",
        MenuItems: []*application.MenuItem{
            {
                Label: "显示/隐藏 Timo",
                Click: func() { toggleWindow() },
            },
            {
                Label: "设置",
                Click: func() { openSettings() },
            },
            application.MenuSeparator(),
            {
                Label: "退出",
                Click: func() { quit() },
            },
        },
    })
}
```

#### 托盘菜单结构

```
Timo (图标)
├── 显示/隐藏 Timo          ← 切换窗口可见性
├── 设置                    ← 打开设置面板
├── ─────────────
├── 优先级: AI > 媒体       ← 灰显，只显示当前状态
├── 主题: 深色              ← 灰显，只显示当前状态
├── ─────────────
└── 退出                    ← 关闭应用
```

#### 跨平台注意事项

| 平台 | 注意点 |
|------|--------|
| **Linux (X11)** | 使用 `libappindicator` 或 `StatusNotifierItem`（Wails v3 内部处理） |
| **Linux (Wayland)** | 许多 DE 不支持传统托盘；需依赖 `StatusNotifierItem` 协议；GNOME 需安装 `AppIndicator` 扩展 |
| **Windows** | 使用 `Shell_NotifyIcon`（Wails v3 内部处理），相对稳定 |

### Phase D：优先级引擎改造

当前 NotchBar.vue 的优先级逻辑是硬编码的：

```vue
<!-- 当前 -->
<template>
  <!-- 1. AI activity (highest priority) -->
  <div v-if="hasAIActivity" class="notch-content ai-mode">...</div>
  <!-- 2. Media (middle) -->
  <div v-else-if="media.hasMedia" class="notch-content media-mode">...</div>
  <!-- 3. Idle (lowest) -->
  <div v-else class="notch-content idle-mode">...</div>
</template>
```

改造为：

```vue
<!-- 改造后 -->
<template>
  <div class="notch-content" :class="modeClass">
    <template v-if="activeMode === 'ai'">
      <!-- AI 内容 -->
    </template>
    <template v-else-if="activeMode === 'media'">
      <!-- 媒体内容 -->
    </template>
    <template v-else-if="activeMode === 'idle'">
      <!-- 空闲内容（根据 idleDisplay 设置决定显示什么）-->
    </template>
    <template v-else>
      <!-- 无内容（当 idleDisplay === 'none' 时）-->
    </template>
  </div>
</template>
```

**计算逻辑（computed）：**

```
activeMode:
  1. 遍历 settings.displayPriority 数组
  2. 对每个模式检查是否「有活动」
     - ai:    notif.state !== ''
     - media: media.hasMedia
  3. 返回第一个有活动的模式
  4. 若都无活动 → "idle"（或 "none" 取决于 idleDisplay 设置）
```

**新增文件（可选）：`frontend/src/composables/useActiveMode.ts`**

将优先级决策逻辑提取到 composable 中，方便测试：

```ts
export function useActiveMode(settings, notif, media) {
    const activeMode = computed(() => {
        for (const mode of settings.displayPriority) {
            switch (mode) {
                case 'ai':    if (notif.state !== '') return 'ai'; break
                case 'media': if (media.hasMedia) return 'media'; break
            }
        }
        return 'idle'
    })
    return { activeMode }
}
```

**空闲显示改造：**

修改 NotchBar.vue 中 idle 部分的渲染，根据 `idleDisplay` 设置决定显示哪些统计项：

```vue
<template v-if="activeMode === 'idle'">
  <div class="idle-content">
    <span v-if="idleDisplay === 'all' || idleDisplay === 'cpu'" class="sys-stat">
      <span class="sys-label">CPU</span>
      <span class="sys-value">{{ cpu }}%</span>
    </span>
    <span v-if="idleDisplay === 'all' || idleDisplay === 'mem'" class="sys-stat">
      <span class="sys-label">MEM</span>
      <span class="sys-value">{{ mem }}G</span>
    </span>
    <span v-if="idleDisplay !== 'none'" class="idle-clock">{{ time }}</span>
  </div>
  <!-- idleDisplay === 'none' 时：只显示时钟或完全隐藏 -->
</template>
```

### Phase F：构建与平台适配

**1. Windows 托盘构建调整**

当前 Windows 构建在 `build/windows/Taskfile.yml` 中：

```yaml
build:native:
    cmds:
        - go build -tags production -trimpath
          -ldflags="-w -s -H windowsgui"
          -o bin/timo.exe .
```

系统托盘在 Windows 上需要 `-H windowsgui`（已存在），不需额外调整。

**2. Linux 托盘依赖**

Linux 托盘可能依赖 `libappindicator-gtk3` 或 `libayatana-appindicator`。需要在 `build/linux/Taskfile.yml` 的 nfpm 配置中添加运行时依赖。

修改 `build/linux/nfpm.yaml`（如果存在）：

```yaml
# 添加运行时依赖
depends:
    - libappindicator-gtk3  # 或 libayatana-appindicator
```

**3. 跨平台配置路径**

```go
// settings.go
func GetSettingsPath() string {
    switch runtime.GOOS {
    case "windows":
        return filepath.Join(os.Getenv("APPDATA"), "timo", "settings.json")
    default: // linux
        return filepath.Join(os.Getenv("HOME"), ".config", "timo", "settings.json")
    }
}
```

## 5. 数据流

### 5.1 设置保存流程

```
SettingsPanel.vue
    ↓ 用户点击「保存」
    ↓ window.runtime.SettingsService.Update(newSettings)
    ↓
SettingsService.Update()
    ├── 更新内存设置
    ├── settings.go: SaveSettings() → 写入 ~/.config/timo/settings.json
    └── mainApp.Event.Emit("settings-updated", settings)
            ↓
        frontend: Events.On("settings-updated")
            ↓
        settings store: 更新响应式状态
            ↓
        NotchBar.vue / island.css: 自动响应
```

### 5.2 设置加载流程（应用启动）

```
main.go: app.Run()
    ↓
SettingsService 初始化 → settings.go: LoadSettings()
    ├── 文件存在 → 读取 JSON
    └── 文件不存在 → 返回默认值 + 写入默认文件
    ↓
mainApp.RegisterService(SettingsService)
    ↓
前端初始化 → SettingsService.Get() → settings store 初始化
    ↓
NotchBar.vue 根据 settings 渲染优先级
```

### 5.3 优先级决策流程

```
定时/事件触发
    ↓
notif store 更新 / media store 更新
    ↓
useActiveMode() 重新计算:
    遍历 settings.displayPriority
    → 检查每个模式是否有活动
    → 返回第一个有活动的模式
    ↓
NotchBar.vue 切换显示
```

## 6. 风险与注意事项

### 🔴 高风险

| 风险 | 影响 | 缓解方案 |
|------|------|----------|
| Wails v3 SystemTray API 不稳定或 Bug | 托盘功能无法正常使用 | 先做最小原型验证；准备方案 B（原生实现 + 条件编译） |
| Linux Wayland 下托盘不可用 | Wayland 用户无托盘 | 不影响现有功能；检测 DE 类型，在 Wayland 下 fallback 到纯窗口模式 |
| CSS 变量替换遗漏导致主题切换后颜色混乱 | 部分 UI 颜色不随主题切换 | 逐文件 grep 硬编码色值，加 CI lint 规则 |

### 🟡 中风险

| 风险 | 影响 | 缓解方案 |
|------|------|----------|
| Settings JSON 并发写入冲突 | 配置文件损坏 | `sync.Mutex` 保护；写入前备份原文件 |
| 前端设置面板与后端设置状态不同步 | 用户看到过期设置 | 所有修改都通过后端 → Event → 前端的单向数据流 |
| 优先级配置导致永远无内容显示（如 `idle: none` 且无活动） | 胶囊栏空白 | 添加最小 fallback：至少显示时钟 |

### 🟢 低风险

| 风险 | 影响 | 缓解方案 |
|------|------|----------|
| 配置文件迁移兼容性 | 旧版本升级后设置丢失 | settings.go 中做字段级别的默认值补全 |
| 主题切换闪烁 | CSS 重新加载瞬间白屏 | 使用 `data-theme` 属性 + CSS 变量，无需重新加载 CSS 文件 |
