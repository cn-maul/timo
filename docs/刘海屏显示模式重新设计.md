# Claude Hooks 刘海屏显示模式重新设计

## 一、现有架构分析

### 1.1 当前 Hook 事件使用情况

Timo 目前使用的 Claude Code hooks：

| Hook 事件 | 用途 | 数据提取 |
|---------|------|---------|
| `UserPromptSubmit` | 检测用户提交新提示 → 进入"运行中"状态 | `prompt`(topic)、`cwd`(workDir) |
| `PostToolUse` | 更新当前工具名称 | `tool_name`/`toolName` |
| `SubagentStart` | 标记子代理启动 | 显示 ⚡ badge |
| `SubagentStop` | 标记子代理结束 | 清除 badge |
| `Notification` | 权限提示/需要关注 | `message` |

**进程监控补充**：`ProcessMonitor` 每秒轮询 `claude`/`reasonix` 进程数，当从 >0 变为 0 时发送 `done` 事件。

### 1.2 当前显示模式

```
┌─────────────────────────────────────────────────────────────┐
│  [Logo] Claude 运行中 ⚡           工具名           1:23  🟢 │
│  ────────────────────────────────────────────────────────── │
│  ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░ │  ← 进度条动画
└─────────────────────────────────────────────────────────────┘

状态指示灯：
- 🟢 绿色闪烁 = running
- 🟡 黄色闪烁 = attention (需要确认)
- 🔴 红色静态 = done
```

### 1.3 问题与改进空间

1. **信息密度不足**：仅显示工具名和计时器，缺少上下文
2. **进度条无意义**：固定动画，无法反映真实进度
3. **缺少阶段区分**：无法区分"正在读取"、"正在编辑"、"正在执行命令"等
4. **子代理信息薄弱**：仅一个 ⚡ 符号，不显示子代理类型和任务描述
5. **未利用丰富 hook 数据**：如 `PreToolUse` 的完整参数、`Stop` 的 `last_assistant_message` 等

## 二、Hook 数据深度挖掘

### 2.1 可利用的 Hook 事件与字段

根据 Claude hooks 文档，以下事件可提供更丰富的显示信息：

#### `PreToolUse` (工具执行前)
```json
{
  "tool_name": "Bash|Edit|Write|Read|Agent...",
  "tool_input": {
    // Bash: command, description, timeout, run_in_background
    // Edit: file_path, old_string, new_string
    // Write: file_path, content
    // Read: file_path, offset, limit
    // Agent: prompt, description, subagent_type
  },
  "tool_use_id": "toolu_01abc",
  "effort": { "level": "low|medium|high|xhigh|max" }
}
```

#### `PostToolUse` (工具执行后)
```json
{
  "tool_name": "...",
  "tool_input": {...},
  "tool_response": {
    // 工具特定结构
  },
  "duration_ms": 123
}
```

#### `SubagentStart`
```json
{
  "agent_id": "agent-abc123",
  "agent_type": "Explore|Plan|general-purpose|自定义..."
}
```

#### `SubagentStop`
```json
{
  "agent_id": "...",
  "agent_type": "...",
  "last_assistant_message": "Analysis complete. Found 3 issues...",
  "background_tasks": [...],
  "session_crons": [...]
}
```

#### `Stop` (会话轮次结束)
```json
{
  "stop_hook_active": false,
  "last_assistant_message": "I've completed the refactoring...",
  "background_tasks": [...],
  "session_crons": [...]
}
```

#### `MessageDisplay` (实时文本流)
```json
{
  "turn_id": "...",
  "message_id": "...",
  "index": 0,
  "final": false,
  "delta": "Here is the plan:\n"
}
```

### 2.2 工具分类与图标映射

| 工具类别 | 工具名 | 建议图标 | 显示要点 |
|---------|--------|---------|---------|
| 文件读取 | Read | 📄 | 文件名(缩短) |
| 文件编辑 | Edit | ✏️ | 文件名 + 操作类型 |
| 文件创建 | Write | 📝 | 文件名 |
| 命令执行 | Bash | ⚙️ | 命令摘要 |
| 子代理 | Agent | 🤖 | 代理类型 + 任务描述 |
| 网络请求 | WebFetch/WebSearch | 🌐 | URL/查询摘要 |
| 用户交互 | AskUserQuestion | ❓ | 问题摘要 |

## 三、重新设计的显示模式

### 3.1 核心设计理念

1. **阶段感知**：区分"思考→读取→编辑→执行→完成"等阶段
2. **上下文丰富**：显示有意义的操作目标(文件名、命令摘要、任务描述)
3. **进度可量化**：基于工具调用计数和时间估算
4. **子代理详情**：显示子代理类型和任务描述
5. **状态流转清晰**：running → attention → done 各有独特视觉

### 3.2 新显示布局


### 3.3 状态详情

#### Running 状态 (绿色)
```
主文本:  [工具图标] [操作目标]
上下文:  [操作详情摘要]
进度:    [已完成工具数/预估总数] 或 [时间进度]
计时:    elapsed time
指示灯:  绿色脉冲发光
```

#### Attention 状态 (黄色)
```
主文本:  ⚠️ 需要确认 / 权限请求
上下文:  [具体请求内容]
指示灯:  黄色脉冲发光
点击:    展开详情 + 操作按钮
```

#### Done 状态 (红色 → 消失)
```
主文本:  ✓ 完成 / [摘要消息]
指示灯:  红色静态 3秒后消失
自动清空延迟: 5秒
```

### 4.2 扩展通知数据结构

```go
type Notification struct {
    Type    string `json:"type"`

    // 现有字段
    Message string `json:"message"`
    Tool    string `json:"tool"`
    WorkDir string `json:"workDir"`
    Topic   string `json:"topic"`

    // 新增字段
    ToolInput   map[string]interface{} `json:"toolInput,omitempty"`   // 工具参数
    ToolOutput  map[string]interface{} `json:"toolOutput,omitempty"`  // 工具结果
    DurationMs  int                    `json:"durationMs,omitempty"`  // 执行耗时
    AgentType   string                 `json:"agentType,omitempty"`   // 子代理类型
    AgentDesc   string                 `json:"agentDesc,omitempty"`   // 子代理任务描述
    AgentResult string                 `json:"agentResult,omitempty"` // 子代理结果摘要
    FinalMessage string                `json:"finalMessage,omitempty"`// Stop 的最后消息
    ToolCount   int                    `json:"toolCount,omitempty"`   // 工具调用计数
    EffortLevel string                 `json:"effortLevel,omitempty"` // 努力级别
}
```

### 4.3 工具名称美化处理

```typescript
// 工具图标映射
const toolIcons: Record<string, string> = {
  Read: '📄',
  Edit: '✏️',
  Write: '📝',
  Bash: '⚙️',
  Agent: '🤖',
  WebFetch: '🌐',
  WebSearch: '🔍',
  AskUserQuestion: '❓',
  Glob: '📁',
  Grep: '🔎',
}

// 工具输入摘要提取
function summarizeToolInput(tool: string, input: any): string {
  switch (tool) {
    case 'Read':
    case 'Edit':
    case 'Write':
      return shortenPath(input.file_path)
    case 'Bash':
      return summarizeCommand(input.command)
    case 'Agent':
      return input.description || truncate(input.prompt, 30)
    default:
      return tool
  }
}
```

## 五、前端状态管理改进

### 5.1 扩展 Notification Store

```typescript
interface ClaudeState {
  phase: 'idle' | 'thinking' | 'reading' | 'editing' | 'executing' | 'subagent' | 'attention' | 'done'

  // 主显示
  primaryText: string      // 工具/任务名称
  secondaryText: string    // 操作详情/上下文
  icon: string             // 当前阶段图标

  // 进度
  toolCount: number        // 已执行工具数
  estimatedTotal: number   // 预估总数(基于历史)
  elapsedTime: number      // elapsed ms

  // 子代理
  subagentType: string
  subagentDesc: string
  subagentResult: string

  // 结束
  finalMessage: string     // Stop 时的摘要

  // 历史(用于进度估算)
  toolHistory: ToolCall[]
}
```

### 5.2 阶段检测逻辑

```typescript
function detectPhase(tool: string, input: any): Phase {
  if (tool === 'Read' || tool === 'Glob' || tool === 'Grep') return 'reading'
  if (tool === 'Edit' || tool === 'Write') return 'editing'
  if (tool === 'Bash') return 'executing'
  if (tool === 'Agent') return 'subagent'
  if (tool === 'WebFetch' || tool === 'WebSearch') return 'reading'
  return 'thinking'
}
```

## 六、实现优先级

### Phase 1: 基础增强 (高优先级)
1. 添加 `PreToolUse` hook，获取完整工具参数
2. 工具图标映射和名称美化
3. 操作目标显示(文件名/命令摘要)

### Phase 2: 进度改进 (中优先级)
1. 工具调用计数和进度条
2. 执行耗时显示
3. 扩展视图(工具列表)

### Phase 3: 子代理增强 (中优先级)
1. 子代理类型图标和名称
2. 任务描述显示
3. 结果摘要显示(SubagentStop)

### Phase 4: 高级功能 (低优先级)
1. `Stop` 事件处理，显示完成摘要
2. `MessageDisplay` 实时文本流(可选)
3. 进度预测算法

## 九、总结

本设计文档基于 Claude hooks 完整文档，深度挖掘了可利用的事件和数据字段，提出了一套更丰富、更有信息量的刘海屏显示方案：

1. **从"仅显示状态"升级为"显示上下文"**：用户能看到 Claude 正在操作什么文件、执行什么命令
2. **从"无意义进度条"升级为"可量化进度"**：基于工具调用计数
3. **从"简单子代理标记"升级为"完整子代理信息"**：类型、任务、结果
4. **保持简洁的同时提供扩展视图**：点击查看更多详情

实现可分阶段推进，优先实现工具图标和操作目标显示，这是性价比最高的改进。
